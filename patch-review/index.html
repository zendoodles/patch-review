Strategy

Scale

  • say what
  • say how

Kinds of reviews

  • update the issue summary
Too much to review! in your comment, say what you reviewed and how (tool used, steps) (scales, because with documented steps to reviewing an issue, others can come by the issue and help review it if you get too busy.) update the issue summary to say what kind of review was done and the comment number

Process

  • cannot unread
  • perfect is enemy of good
You cannot un-read the issue, sometimes read the patch first to see if it makes sense Perfect is the enemy of good ... for nits, just fix the nits, do review separate You will find a process you like, these demos give a starting point and inspiration

Performance

  • Don't guess
  • profile!
read the core gate, document, don't guess profile, yourself or mark it needs profiling with tag and contributor task [fill in detail?] sql queries, caches, memory usage, cpu, system calls, front-end contributor task for profiling

Accessibility

  • bug
  • WCAG
  • ATAG
  • text color contrast
  • form fields are labeled
  • keyboard-usable (javascript)
it's a bug conforms to WCAG 2.0 ATAG 2.0 run a tool that gives output text color contrast form fields are labeled keyboard-usable (javascript)

Usability

  • UI changes need usability review
  • before and after screenshots
UI changes undergo usability review. provide before and after screenshots acknowlege a change in the UI, tag usabilty, for hard things: needs usability review

Testing

  • check coverage
  • ask for new tests
  • verify tests-only fails
check test coverage & ensure tests pass ask for add new tests verify uploaded a test case that fails (for bugs) pink green manually test and provide screenshots watch for tests being changed, ask why fix bug and prove test coverage, so bug does not sneek in next time. steps are given to manually test javascript (there are not yet automated tests for javascript)
